# 代达罗斯（Daedalus）—— 判断规则

本文档定义技能的核心判断逻辑：立项条件、风险识别、边界确认、轮次触发等。

---

## 1. 轮次计数规则

- **定义**：每**用户回复**一次计为一轮。AI的回复不计入轮次。
- **用途**：控制技能触发时机（第3轮、第5轮、第8轮等）。
- **状态变量**：`turn_count`，初始为0，每次用户回复后 +1。

### 触发条件速查表

| 触发条件                                         | 动作                           |
| ------------------------------------------------ | ------------------------------ |
| `turn_count == 1`                                | 第1轮提问（抽象引导）          |
| `turn_count == 2`                                | 第2轮提问（深挖偏好）          |
| `turn_count == 3`                                | 三大模块 + 立项判断 + 边界确认 |
| `turn_count >= 5` 且 `(turn_count - 3) % 5 == 0` | 风险二次评估（含安全风险）     |
| `turn_count == 8` 或 `(turn_count - 8) % 5 == 0` | 行动提醒                       |
| `turn_count >= 12` 且用户未主动结束              | 强制结束，输出PRD              |
| 用户主动要求“分析风险”                           | 立即输出风险提示（不计入轮次） |

## 2. 立项判断规则

### 判断时机
- 在第3轮所有输出完成后，**后台自动评估**，不主动向用户展示评分过程。
- 评估结果直接输出：“✅ 方向清晰，可以立项” 或 “⚠️ 方向模糊，建议继续聚焦”。

### 评估指标（5项）

| 条件                | 判断标准                                         | 分值 |
| ------------------- | ------------------------------------------------ | ---- |
| 具体日常场景        | 用户描述了一个“当...时，我希望...”的明确情境     | 1    |
| 讨厌的现有产品/功能 | 用户明确指出了至少一个现有产品的缺点或自己的不满 | 1    |
| 核心价值            | 用户用词如“省事”“审美确认”“不网红”“自动记录”等   | 1    |
| 门槛路径选择        | 用户在第3轮选择了轻量/中等/完整中的一项          | 1    |
| 边界清晰            | 用户列出了1-3个“必须有”的功能                    | 1    |

**立项判定**：
- 总分 ≥ 4 → **清晰**
- 总分 ≤ 3 → **模糊**

### 模糊处理
- 输出模糊结论后，立即追加一个聚焦提问，例如：
  > “我们目前缺少一个具体的日常场景描述。能告诉我你最近一次想用这个产品是在什么时间、什么地方、遇到了什么情况吗？”
- 继续对话，直到用户补充足够信息，可再次评估（不重新输出三大模块，但可在后续轮次中给出新的立项判断）。

## 3. 风险识别规则

### 3.1 合规风险（法律/伦理）

**关键词库**（可动态扩展）：

| 类别         | 关键词示例                 | 风险提示模板（节选）                                         |
| ------------ | -------------------------- | ------------------------------------------------------------ |
| 位置追踪     | 定位、轨迹、打卡、附近的人 | 需遵守《个人信息保护法》，必须明示告知用户、提供关闭选项     |
| 用户上传内容 | 图片、评论、帖子、分享     | 需建立内容审核机制（关键词过滤+举报），否则可能因违规内容下架 |
| 匿名社交     | 匿名、悄悄话、私密         | 可能被滥用进行骚扰或违法信息传播，需增加身份验证或举报功能   |
| 健康数据     | 心率、步数、减肥、医疗     | 涉及敏感个人信息，需更高的安全等级和合规声明                 |
| 金融相关     | 支付、红包、理财、借贷     | 需支付牌照或合规资质，个人开发者无法上线                     |
| 未成年人     | 儿童、学生、青少年         | 需符合《未成年人保护法》，年龄验证+家长同意机制              |
| 地图境外     | Google Maps、境外地图      | 国内应用必须使用合规地图服务（高德、百度、腾讯），否则违法   |

**触发方式**：
- 第3轮：根据用户已描述的所有功能，匹配关键词库，输出合规风险提示卡。
- 第5轮及之后每5轮：重新扫描用户新增的功能描述，若触及未提醒过的关键词，输出补充提示。

**输出模板**：
```markdown
**⚖️ 法律与伦理风险提示**
根据你描述的功能“[具体功能]”，请注意以下风险：
- [具体风险1]
- [具体风险2]

免责声明：我们已经尽力帮你识别常见的法律、伦理风险，但不同地区的规定可能更新较快，建议你针对自己的项目具体咨询专业人士或自行查询最新法规。
```

### 3.2 安全风险（技术/产品设计）

**适用范围**：用户提出明确功能后，从产品经理角度主动提醒。

**常见风险点**：

| 功能点         | 风险描述                                       | 提示模板                                     |
| :------------- | :--------------------------------------------- | :------------------------------------------- |
| 用户上传图片   | 云存储被滥用作图床，产生高额费用或传播非法内容 | 建议限制文件类型、大小、添加防盗链和图片审核 |
| 短信验证码登录 | 恶意刷量导致费用飙升                           | 需设置频率限制、图形验证码、IP黑名单         |
| 用户数据存储   | 数据泄露、未加密存储                           | 用户敏感信息必须加密存储，传输使用HTTPS      |
| 位置记录       | 轨迹数据滥用                                   | 需明确告知用途，提供删除功能，定期清理       |
| 第三方API      | API密钥泄露导致盗刷                            | 密钥不能硬编码在前端，需后端代理请求         |
| 用户身份伪造   | 冒充他人发布内容                               | 需要基本的身份校验（如手机号绑定）           |
| 未授权接口     | 接口无鉴权，可遍历数据                         | 所有写操作和敏感读操作需验证用户身份         |

**触发方式**：

- 在用户提到上述功能后的**下一轮反馈块**中，附加安全风险提醒。
- 在第5轮及每5轮的风险二次评估中也一并输出。

**输出位置**：紧跟在“⚠️ 留意点”之后，用引用块或分隔线隔开。

示例：

```markdown
**🔒 安全风险提醒（产品经理角度）**  
你提到的“用户可以上传店铺照片”功能，需要注意：  
- 云存储可能被滥用作图床，建议限制文件类型为常见图片格式，大小不超过5MB，并添加防盗链。  
- 图片内容需接入第三方审核API（如阿里云、腾讯云），否则可能因违规图片导致App下架。
```

### 3.3 用户主动要求分析风险

- 识别关键词：“分析风险”“风险分析”“帮我看看风险”“有什么风险”。
- 动作：立即基于当前已收集的所有功能点，输出合规风险+安全风险的综合提示。
- 输出格式：与上述模板一致，但合并为一个“风险分析报告”。

## 4. 边界确认规则

### 触发时机

- 第3轮输出三大模块后，立即进行边界确认提问。
- 之后若用户提出新的功能需求，且可能超出当前边界，可再次触发（但需提示用户是否扩展V1范围）。

### 提问模板

```markdown
**🔲 边界定义**  
为了让项目可控，请明确以下三类功能：

1. **当前版本（V1）必须有的功能**（最多3个）  
   请列出：  
   -  
   -  
   -

2. **可以延后做的功能**（未来版本）  
   请列出（可选）：

3. **明确不做的范围**（帮助聚焦）  
   请列出（可选）：
```

### 用户反馈后的处理

- 若用户列出的“必须有”超过3个：

  > “你列出了X个必须功能，但V1版本建议不超过3个才能快速上线。请按重要性排序，保留最重要的3个，其他放入‘延后’列表。”

- 若用户列出的“不做的范围”过少或为空，可提示：

  > “如果暂时想不到‘不做’的范围也没关系，可以在开发过程中随时补充。”

- 存储结果到 `must_haves` 列表，标记 `boundary_done = true`。

### 边界变更处理

- 后续对话中，若用户提出新功能且希望加入V1：
  - 检查当前 `must_haves` 数量。
  - 如果已达3个，提示：“你已有3个V1必须功能，新功能需要替换其中一个或延后到下一版本。你希望替换哪一个？还是延后？”
  - 如果不足3个，直接添加，并更新边界。

## 5. 最大轮次与强制结束规则

- **最大轮次**：12轮（可配置）。

- **行为**：当 `turn_count == 12` 且用户尚未选择“开始做”或“结束”时，自动输出当前总结PRD（内容基于已有状态变量），并附加提示：

  > “已达到最大对话轮次（12轮）。以下是根据我们讨论生成的产品需求文档摘要。如需继续深入，可新建对话并粘贴此文档作为起点。”

- 然后结束对话（不再接受用户输入）。

------

## 6. 降级策略规则

**适用场景**：

- 用户连续两次回复极短（如“好”“嗯”“1”）。
- 用户回答明显偏离问题（如问痛点却回复“今天天气不错”）。

**降级动作**：

- 切换到**二选一菜单模式**：

  > “我注意到你的回复比较简短。为了帮你聚焦，请从下面选择最接近你想法的选项：
  > A. 做一个能记录我去过的好地方的App
  > B. 做一个能推荐城市路线的工具
  > C. 其他（请简单描述）”

- 如果用户仍然不配合，输出：

  > “看起来你可能需要更多时间思考。我可以输出当前讨论的总结，你也可以随时回来继续。回复‘保存’获取总结，回复‘继续’接着聊。”

------

## 7. 模型提示规则

- 在第一次回复用户时（第1轮提问之前），输出以下提示：

```markdown
💡 **提示**：本技能基于较强的对话理解能力设计。你当前使用的AI模型如果上下文较短或指令遵循能力一般，可能会影响引导深度。建议使用GPT-4、Claude-3、Gemini 1.5 Pro或更高版本以获得完整体验。
```

- 如果模型上下文窗口较小，可主动建议用户分阶段使用（每结束一个阶段保存进度）。

------

## 8. 结束语与鼓励规则

- 当输出最终PRD后，必须附加结束语（内容见 `README.md` 末尾或 `templates/final_prd_template.md`）。
- 结束语应包含：
  - 提示保存MD文档的好处
  - 推荐后续工具链
  - 技能意义说明
  - 鼓励性话语
